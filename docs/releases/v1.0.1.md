# Release Notes - v1.0.1

## Version
`1.0.1`

## Summary
This version adds public session registration, improves player-related API responses, automates payment creation more reliably, and expands project documentation.

## New Features

### Public session registration
A new public API was added so anyone can register into a session using:
- name
- gender
- phone number

Endpoint:
- `POST /api/session/{id}/register`

Behavior:
- checks whether the player already exists by phone number
- searches within the session owner's tenant data
- creates a new member automatically if the phone number does not exist
- registers the member into the session
- prevents duplicate registration in the same session
- prevents registration into overlapping active sessions
- auto-generates player payment if the session is already `OnGoing`

### Session match management
A full match management feature was added for sessions.

New endpoints:
- `GET /api/session/{sessionId}/matches`
- `GET /api/session/{sessionId}/matches/{matchId}`
- `POST /api/session/{sessionId}/matches`
- `PUT /api/session/{sessionId}/matches/{matchId}`
- `DELETE /api/session/{sessionId}/matches/{matchId}`

Behavior:
- each session can have multiple matches
- each match has team A and team B
- each team supports 1 to 2 players
- only players already registered in the session can be used in a match
- scores and winner are stored for each match
- winner validation matches the submitted score
- session detail now includes match information

### Elo reward system
Elo points are automatically applied to player rankings when a match result is set.

Formula:
- base reward: 25 points per player
- each 200-point Elo gap between the two teams adds 10% to the reward
  - gap 0–199: 25 pts
  - gap 200–399: 28 pts
  - gap 400–599: 30 pts
  - and so on
- the winning team gains the reward, the losing team loses the same amount
- a draw results in no Elo change
- the player tier (`RankingId`) is re-evaluated automatically after each Elo adjustment

Reversal behavior:
- updating or deleting a match automatically reverses previously applied Elo before applying the new result
- the actual Elo change per player is stored on `SessionMatchPlayer.EloChange` to enable exact reversal

Model changes:
- `SessionMatch.IsEloApplied` — tracks whether Elo has been applied for a match
- `SessionMatchPlayer.EloChange` — stores the Elo delta applied to each player

## Improvements

### Player API responses now include level and Elo point
Player-related responses were enhanced to expose:
- `Level`
- `EloPoint`

This applies to:
- `SessionPlayerReadDto`
- player items returned by session detail

### Automatic player payment generation
Player payments are now created automatically when:
- a session changes to `OnGoing`
- a player is added after the session has already started
- session prices are set for an ongoing session

This reduces manual payment setup steps for organizers.

### Payment API cleanup
The unused public endpoint for manual bulk player-payment generation was removed.

Removed endpoint:
- `POST /api/payment/session/{sessionId}/generate`

### Contact lookup improvements
Member contact lookup was improved by:
- normalizing input with trimming
- handling stored contact values with extra whitespace
- using clearer ignore-filter lookup behavior where required

This helps member lookup return more reliable results.

### Member detail debt grouping
The member detail API now includes grouped debt information in `UnpaidByUser`.

Behavior:
- unpaid session debt is grouped by session owner `UserId`
- the response includes owner and bank information for each group
- the shape matches the grouped debt data used in the lookup API
- authenticated owners only see debt entries for their own sessions in owner-facing member detail APIs

### Member match statistics
Member APIs now include match statistics based on recorded session matches.

Fields added:
- `Wins`
- `Losses`
- `Draws`
- `WinRate`

Behavior:
- owner-facing member detail returns stats calculated only from that owner's sessions
- player lookup returns overall stats across all recorded matches
- only matches with a non-pending result are counted
- win rate is returned as a percentage rounded to 2 decimal places
