# Release Notes — v1.1.7

## Summary
Refines the payment confirmation flow to require explicit Owner approval before a payment is marked as Paid (BCM-110).

> **Terminology**
> - **User** — a player who does **not** need authentication (public-facing actions).
> - **Owner** — a player who **requires** authentication (JWT). Manages sessions, members, and payments.

## Changes

### Two-step Payment Confirmation Flow
```
Before:  User/Owner → POST /confirm → Paid  (no approval step)

After:   Owner → POST /confirm → ConfirmationPending + Owner notified via notification
         Owner → POST /approve → Paid
```

### New Enum Values
- `PaymentStatus.ConfirmationPending = 3`
- `SessionPlayerStatus.ConfirmationPending = 4`

### Updated Endpoint — Owner only
- `POST /api/payment/session-player/{id}/confirm` — marks the payment as `ConfirmationPending` and sends a `PaymentRecorded` notification to the Owner.

### New Endpoint — Owner only
- `POST /api/payment/session-player/{id}/approve` — Owner approves the pending payment; sets `PlayerPayment.PaidStatus = Paid`, `SessionPlayer.Status = Paid`, and records `PaidAt`.

## Migration
